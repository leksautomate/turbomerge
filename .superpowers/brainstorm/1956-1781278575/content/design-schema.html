Design: Database Schema

Section 2 of 5 — Two tables, Prisma ORM, PostgreSQL

projects
id             String   @id @default(uuid())
title          String   // required, shown on dashboard
description    String?
media_type     Enum     // IMAGE | VIDEO
provider       Enum     // RUNWARE | VERTEX
settings       Json     // model, quality, format, aspectRatio…
status         Enum     // DRAFT | GENERATING | PAUSED | COMPLETED | CANCELLED
total_jobs     Int      @default(0)
completed_jobs Int      @default(0)
failed_jobs    Int      @default(0)
created_at     DateTime @default(now())
updated_at     DateTime @updatedAt
jobs           Job[]
jobs
id                  String   @id @default(uuid())
project_id         String   // FK → projects.id
prompt             String
status             Enum     // PENDING | GENERATING | COMPLETED | FAILED | CANCELLED
progress           Int      @default(0)  // 0–100, used for video
output_path        String?  // local disk path when done
error_message      String?
retry_count        Int      @default(0)
vertex_operation_id String? // video only — survives restarts
created_at         DateTime @default(now())
updated_at         DateTime @updatedAt
project            Project @relation(…)
Key decisions
  • settings is JSON — model, quality, format, aspectRatio all live here per project
  • vertex_operation_id is stored so video polling resumes after a restart
  • output_path is a relative path like uploads/images/<projectId>/1.png
  • Project counters (total_jobs, completed_jobs, failed_jobs) update atomically when a job finishes
  • No separate users table — single user app for now
