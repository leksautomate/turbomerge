Architecture Approaches

Three ways to structure the bulk generation system — pick the one that fits best.

A

Next.js + pg-boss + Worker Script Recommended

Next.js handles the UI and API routes. A separate worker.ts script runs alongside it (started with concurrently) and processes jobs via pg-boss. One repo, two processes, one npm run dev.

Browser → Next.js API routes → PostgreSQL (pg-boss queue)
                                  ↓
                        worker.ts (always running)
                                  ↓
            Runware API       Vertex AI Veo
                                  ↓
                  uploads/ (local disk)

Pros

  • Built-in retries & concurrency
  • Jobs survive server restarts
  • One repo, simple setup

Cons

  • Two processes to start
  • pg-boss adds a dependency
B

Next.js + Raw DB Polling Worker

Skip pg-boss entirely. The worker just polls the jobs table every few seconds: grab a pending job, process it, mark it done. No extra library — just SQL.

Browser → Next.js API routes → PostgreSQL (jobs table)
                                  ↓ poll every 3s
                        worker.ts (setInterval loop)
                                  ↓
            Runware API       Vertex AI Veo

Pros

  • Zero extra libraries
  • Easiest to read & debug
  • Full control

Cons

  • Manual retry logic
  • Race conditions if running 2+ workers
  • More code to write
C

Next.js Inline Processing (avoid)

Process images/videos directly inside Next.js API route handlers — no separate worker, no queue.

Pros

  • Simplest possible setup
  • No extra files

Cons

  • API routes time out — kills long video jobs
  • Bulk of 100+ images will crash
  • No pause/resume/retry possible
